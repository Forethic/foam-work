# 分阶段系统选型策略 - 宜搭过渡方案

**文档编号**：APP-DEC-002  
**版本**：v1.0  
**创建日期**：2026年1月5日  
**策略定位**：成本优先、快速交付、渐进升级

---

## 一、策略概述

### 1.1 核心理念

**"先用宜搭快速搭建，满足基本需求，待业务成熟后再升级为专业系统"**

这是一种**精益启动、渐进成熟**的策略：

- ✅ **第一阶段（0-12个月）**：用宜搭快速搭建，验证业务流程，成本极低
- ✅ **第二阶段（12-24个月）**：业务复杂度提升后，逐步升级为专业系统
- ✅ **第三阶段（24-36个月）**：形成完整的企业级系统架构

### 1.2 成本对比

| 策略                         | 第一年投资 | 第二年投资 | 3年总成本 | 优势                             |
| ---------------------------- | ---------- | ---------- | --------- | -------------------------------- |
| **一步到位（专业系统）**     | 290万      | 60万       | 640万     | 功能完整，但初期投入大           |
| **宜搭过渡策略（超精益版）** | **8万**    | 80万       | **238万** | 节省402万（63%），现金流优势极大 |

**节省成本**：约**402万**（63%成本优化）  
**现金流优势**：第一年仅需8万，节省282万现金流

- ❌ 复杂算法（APS排程、MRP计算、成本核算）
- ❌ 极高性能要求（秒级响应、百万级并发）

**系统清单**（0个，第一年全部暂缓）：

### 2.4 公司专用系统（独立部署，数据隔离）

**定义**：涉及核心技术保密或业务模式差异极大的系统，各公司部署**独立实例**。

**核心原则**：

- ❗**数据完全隔离**：A公司和B公司拥有独立的系统实例和数据库
- ❗**技术保密优先**：研发数据、配方、工艺等敏感信息严格隔离
- ✅ **可选择性集成**：通过受控的API接口，实现必要的数据交换（如技术转移时）

#### **📋 公司专用系统清单**

**1. ELN电子实验记录本** 🔒

**为什么必须独立部署？**

- ✅ 研发保密要求极高：配方、工艺路线、实验数据属于核心机密
- ✅ A公司：新产品研发实验（前沿技术、专利布局）
- ✅ B公司：工艺改进实验（生产优化、成本降低）
- ✅ 避免数据泄露：即使是集团内部，也需要严格隔离

**宜搭方案（第一年）**：

- **A公司ELN**：独立宜搭应用，仅A公司人员可访问
- **B公司ELN**：独立宜搭应用，仅B公司人员可访问
- **成本**：0元（宜搭应用数量不限）

**专业系统升级（第二年起）**：

- **A公司ELN**：慧云ELN（专业版，符合GLP），预算12万
- **B公司ELN**：慧云ELN（标准版），预算8万
- **升级时机**：需要FDA认证、CNAS认证、专利诉讼证据或仪器集成时

**技术转移场景处理**：

- 当A公司研发成果需要转移到B公司生产时
- 通过**受控的导出/导入**机制：A公司导出脱敏后的工艺包 → B公司导入到自己的ELN
- 或通过**临时授权**：B公司指定人员获得A公司特定项目的只读权限（审计留痕）

**⭐ 公司专用 vs 集团共享对比**

| 维度 | 集团共享系统（如PLM、LIMS） | 公司专用系统（如ELN、MES） |
|-----|---------------------------|---------------------------|
| **数据隔离** | 逻辑隔离（权限控制） | 物理隔离（独立实例） |

| **典型系统** | PLM、LIMS、QMS、CRM、SRM | ELN、MES、核心研发工具 |

---

**⭐ 重点说明 - 为什么LIMS可以共享，但ELN必须独立？**

**LIMS实验室信息管理**（可共享）：

- 主要管理质量检测数据（常规分析、合格判定）
- 检测标准相对公开（国标、行标）
- 保密级别：中等（检测方法和结果不是核心机密）
- 集团级LIMS可统一检测标准，降低成本

**ELN电子实验记录本**（必须独立）：

- 记录研发过程数据（配方、工艺摸索、失败记录）
- 涉及核心技术路线和专利布局
- 保密级别：极高（研发数据是企业核心竞争力）
- 即使集团内部，也需要严格的知识产权保护

**策略**：根据实际使用情况，选择3-5个宜搭应用升级为专业系统

**升级优先级评估标准**：

| 评分项     | 权重 | 说明                      |
| ---------- | ---- | ------------------------- |
| 用户抱怨度 | 40%  | 用户反馈宜搭性能/功能不足 |
| 数据量增长 | 30%  | 接近5万条上限             |
| 业务复杂度 | 20%  | 业务规则复杂化            |
| ROI价值    | 10%  | 升级后带来的效益          |

**典型升级路径示例**：

#### 场景1：客户量快速增长

```txt
M13：CRM宜搭应用客户数突破500个，销售团队扩张到25人
  → 性能开始下降，销售漏斗报表加载慢
  → 决策：升级为专业CRM（纷享销客），预算15万
  → M14-M15：实施专业CRM，数据从宜搭迁移
  → M16：上线，销售效率提升30%
```

#### 场景2：产品线复杂化

```txt
M18：产品线从20个增长到60个，配方管理复杂度大幅提升
  → 宜搭PLM已有800+配方，版本管理混乱
  → 决策：升级为专业PLM（PTC Windchill），预算40万
  → M19-M21：实施专业PLM
  → M22：上线
```

#### 场景3：仓储精细化

```txt
M20：B公司仓库扩建，SKU从300增至1200，需要PDA扫码
  → 宜搭WMS无法满足
  → 决策：升级为专业WMS（富勒），预算20万
  → M21-M23：实施专业WMS
  → M24：上线
```

## 五、数据迁移策略

### 5.1 宜搭 → 专业系统数据迁移

**标准流程**：

```txt
1. 导出宜搭数据（Excel/API）
2. 数据清洗与映射
3. 导入专业系统（批量导入工具）
4. 数据对账验证
5. 双轨运行2周
6. 正式切换
7. 宜搭应用归档
```

**关键要点**：

- ✅ 宜搭设计时预留"导出"功能
- ✅ 数据结构设计尽量与专业系统对齐
- ✅ 保留历史数据（只读模式）

### 5.2 数据结构设计原则

在宜搭中设计数据表时，就要考虑未来升级：

**示例：物料主数据表**

```json
{
  "material_id": "M0001",          // 物料编码（唯一键）
  "material_name": "聚氨酯A-36",   // 物料名称
  "category": "原料",              // 分类
  "unit": "kg",                    // 单位
  "safety_stock": 100,             // 安全库存
  "company_id": "A",               // 所属公司（预留字段）
  "status": "active",              // 状态
  "created_by": "张三",
  "created_at": "2026-01-05"
}
```

**设计要点**：

- 使用与专业系统兼容的字段名
- 预留扩展字段（company_id, status等）
- 避免使用宜搭特有的复杂控件（如子表单）
