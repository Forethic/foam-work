# 文档命名规范 (V1.3 最终版)

## 1. 目的

为统一知识库内所有文档的命名与存放标准，提高文档的可识别性、可搜索性和可维护性，特制定本规范。

## 2. 通用规则

* **语言**: 统一使用中文，特殊技术术语可使用英文。
* **分隔符**: 各组成部分之间，统一使用半角减号 `-` 分隔。
* **日期格式**: 统一使用8位数字格式 `YYYYMMDD`。
* **版本管理**: 不在文件名中体现版本号，利用WPS自带的版本历史功能。
* **专业缩写**: 在描述文档类型时，推荐使用业界通用的、广为人知的专业缩写词（如 `SRS`, `BRD`, `SOW`），以保证简洁和专业。团队内部应对常用缩写有统一认知，具体可参见本文末尾的**附录**。

## 3. 命名与存放的核心原则

* **特异性优先**: 当一份文档同时符合多种类型时，**内容属性**的优先级高于**形式属性**。例如，一份项目会议纪要，其“项目”属性优先于“会议”属性，应使用 `PROJ` 前缀。
* **存放位置明确**: 每种类型的文档，都必须存放在其指定的目录下，以保证知识库结构的稳定性。

## 4. 按文档类型命名与存放

### 4.1. `SOP` - 标准作业程序类

* **格式**: `SOP-部门-流水号-简短描述`
* **存放位置**: `30_标准作业程序/`
* **示例**: `SOP-IT-001-新员工入职IT配置流程.docx`

### 4.2. `PROJ` - 项目文档类 (高优先级)

* **格式**: `PROJ-项目代号-文档描述`
* **存放位置**: `40_项目文档/` 下对应的项目文件夹内。
* **示例**: `PROJ-CRM2025-项目启动会纪要-20251112.docx`

### 4.3. `SOL` - 采购与解决方案类 (高优先级)

* **格式**: `SOL-采购主题-供应商-文档类型-日期`
* **存放位置**: `60_采购与解决方案/` 下对应的采购项目文件夹内。
* **示例**: `SOL-低代码平台选型-通用SRS-20251111.docx`

### 4.4. `TECH` - 技术文档类

* **格式**: `TECH-技术领域-具体主题`
* **存放位置**: `10_IT基础设施/` 或 `20_系统与应用/` 下对应的技术/系统目录中。
* **说明**: 用于记录**客观的、权威的**技术事实或操作指南（“How-to”）。
* **示例**: `TECH-网络-核心交换机端口镜像配置指南.docx`

### 4.5. `MEET` - 通用会议纪要类

* **格式**: `MEET-会议名称-日期`
* **存放位置**: 根据会议内容，存放在最相关的目录下。
* **说明**: 仅用于**不属于**任何特定项目、采购或SOP的**常规性**会议。
* **示例**: `MEET-IT部门月度沟通会-202511.docx`

### 4.6. `TEAM` - 团队建设类

* **格式**: `TEAM-文档描述`
* **存放位置**: `70_团队建设与管理/`
* **示例**: `TEAM-IT部门技能矩阵.xlsx`

### 4.7. `SHARE` - 知识分享类

* **格式**: `SHARE-日期-分享人-分享主题`
* **存放位置**: `50_知识分享/` 下对应的主题文件夹内。
* **说明**: 用于存放带有**个人经验、观点或思考**的培训材料或分享记录（“Why & Experience”）。
* **示例**: `SHARE-20251110-张三-MySQL主从搭建的实践与思考.pptx`

## 5. 附录：常用专业缩写词对照表

| 缩写  | 英文全称                       | 中文全称           | 使用场景/说明                                |
| :---- | :----------------------------- | :----------------- | :------------------------------------------- |
| **BRD** | Business Requirements Document | 商业需求文档       | 描述项目的商业价值、市场机会和高阶目标。     |
| **MRD** | Market Requirements Document   | 市场需求文档       | 从市场角度分析用户、场景，定义产品需满足的需求。 |
| **PRD** | Product Requirements Document  | 产品需求文档       | 详细描述产品的功能、界面、交互和流程。       |
| **SRS** | Software Requirements Spec.    | 软件需求规格说明书 | (同PRD) 更偏向软件工程，是开发和测试的直接依据。 |
| **FSD** | Functional Specifications Doc. | 功能规格说明书     | 详细描述某个具体功能点的逻辑和规则。         |
| **SOW** | Statement of Work              | 工作说明书         | (采购/外包) 详细定义项目范围、交付物、时间表。 |
| **RFP** | Request for Proposal           | 方案建议邀请书     | (采购) 向供应商发出的、邀请其提供解决方案的正式文件。 |
| **FAQ** | Frequently Asked Questions     | 常见问题解答       | 用于整理和回答用户或内部的常见问题。         |
| **KB**  | Knowledge Base                 | 知识库             | 对某一领域知识的系统性整理。                 |
| **POC** | Proof of Concept               | 概念验证           | 为验证某个技术或方案的可行性而进行的最小化实现。 |
| **UAT** | User Acceptance Test           | 用户验收测试       | 由最终用户进行的、确认系统是否满足需求的测试。 |
